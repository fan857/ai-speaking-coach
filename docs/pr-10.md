# PR 10：接入 Qwen3.5-Omni-Realtime 实时语音对话

## 本 PR 做了什么

- 新增后端 `WebSocket /api/realtime/qwen`，代理连接阿里云百炼 Qwen3.5-Omni-Realtime。
- 前端新增“开始 Qwen 实时语音对话”入口。
- 前端将旧“开始录音”升级为“录一句并识别”，使用 Qwen 实时 ASR 转写单句。
- 前端采集麦克风音频，转换为 PCM16 后持续发送到后端。
- 后端隐藏 `DASHSCOPE_API_KEY`，前端不直接暴露第三方模型密钥。
- 前端播放模型返回的实时 PCM 音频，并展示实时识别文本和回复文本。
- 保留原有 DeepSeek 完整反馈、浏览器语音识别、低延迟播放队列和本地 fallback。
- 下线主界面中的旧“低延迟流式发送”慢路径，把实时对话统一交给 Qwen3.5-Omni-Realtime。
- 将原文本提交按钮定位为“获取完整纠错评分”，用于慢但更完整的课后/逐句分析。

## 为什么这样做

题目要求关注实时语音对话、端到端流畅性和延迟。原来的浏览器识别 + DeepSeek + 浏览器朗读是三段式链路，真实语音对话延迟较高。

Qwen3.5-Omni-Realtime 是实时音频输入和音频输出模型，更适合作为语音对话底座。本项目仍保留自己的训练流程、模式切换、反馈面板和 fallback 机制，避免只做单纯 API 套壳。

## 核心实现思路

- 后端通过 `websockets` 连接 DashScope：
  - `wss://dashscope.aliyuncs.com/api-ws/v1/realtime?model=qwen3.5-omni-plus-realtime`
- 后端读取环境变量：
  - `DASHSCOPE_API_KEY`
  - `DASHSCOPE_REALTIME_MODEL`
  - `DASHSCOPE_REALTIME_VOICE`
- 后端根据当前练习场景生成实时对话 instructions。
- 前端使用 `getUserMedia` 获取麦克风音频。
- 前端将浏览器音频下采样为 16k PCM16，并通过本地 WebSocket 发送 `input_audio_buffer.append`。
- 后端把前端事件转发给 DashScope，把 DashScope 事件转发回前端。
- 前端处理 `response.audio.delta` 并播放 PCM 音频。
- 前端展示实时识别文本和模型回复文本，方便 Demo 讲解。
- 逐句反馈模式：Qwen ASR 识别一句话后自动提交 DeepSeek 生成纠错评分。
- 沉浸对话模式：Qwen3.5-Omni-Realtime 直接进行实时语音对话，结束后再做总结。

## 如何测试

1. 在 `backend/.env` 中配置：

```text
DASHSCOPE_API_KEY=你的阿里云百炼 API Key
DASHSCOPE_REALTIME_MODEL=qwen3.5-omni-plus-realtime
DASHSCOPE_REALTIME_VOICE=Tina
```

2. 安装依赖：

```bash
npm run install:all
```

3. 启动后端：

```bash
npm run dev:backend
```

4. 启动前端：

```bash
npm run dev:frontend
```

5. 打开：

```text
http://localhost:5173
```

6. 点击“开始 Qwen 实时语音对话”，允许麦克风权限。

7. 用英文说一句话，确认：

- 页面状态显示 Qwen 实时模型已连接。
- “你说的话”区域出现实时识别结果。
- “实时回复”区域出现模型回复文本。
- 浏览器能播放模型返回的实时音频。
- 点击“停止 Qwen 实时对话”后麦克风和 WebSocket 会关闭。

8. 不配置 `DASHSCOPE_API_KEY` 时，确认原有 DeepSeek / mock / 浏览器识别流程仍可使用。
