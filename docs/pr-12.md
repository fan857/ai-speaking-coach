# PR 12：修正逐句反馈链路与真实来源展示

## 本 PR 做了什么

- 逐句反馈模式下移除沉浸式 Qwen 实时语音对话面板，只保留单句练习入口。
- 将“录一句并识别”升级为“录一句并对话”：Qwen Realtime 负责实时识别用户语音，并先给出一句自然英文接话。
- “获取完整纠错评分”按钮在用户还没有说过一句话前保持禁用；说完并获得 AI 接话后，用户可主动点击生成完整纠错评分。
- 移除逐句反馈里的“发送并继续对话”按钮，避免和语音练习入口重复。
- 明确展示当前回复来源：Qwen Realtime、DeepSeek、快速本地兜底或 mock，避免误以为兜底回复是真实模型回复。
- 优化餐厅场景 mock 快速回复，避免多轮对话里反复返回同一句追问。
- 沉浸式对话模式继续保留 Qwen3.5-Omni-Realtime 实时语音入口，并在结束后统一生成总结。
- DeepSeek 调用从 LangChain 封装改为后端直接请求 DeepSeek OpenAI-compatible Chat Completions API，减少依赖层导致的失败。
- 保留默认 DeepSeek 模型 `deepseek-v4-flash`，并补充 `DEEPSEEK_BASE_URL` 配置。
- 真实 AI 请求失败时返回更具体的 warning，帮助定位 Key、模型名、网络或额度问题。

## 为什么这样做

逐句反馈和沉浸式对话是两种不同训练路径。逐句反馈需要低延迟地“说一句、AI 接一句”，但完整纠错评分可以稍后由用户主动触发。这样既能让对话不空等，也能避免每一轮都被评分打断。

同时，页面必须诚实展示回复来源。如果走了 mock 或本地兜底，用户和评委都应该能看出来；如果走 Qwen Realtime，也应该明确标出这是实时语音模型先回复。

## 核心实现思路

- 前端根据 `isImmersiveMode` 控制 Qwen 实时语音面板是否展示。
- 逐句反馈模式点击“录一句并对话”后，前端连接 Qwen Realtime WebSocket，并使用 `mode=feedback` 让模型完成“识别 + 简短接话”。
- Qwen Realtime 的逐句提示词只要求自然接话，不输出纠错、评分和学习建议。
- 前端保存最近一条用户发言为 `feedbackCandidate`，只有存在待评分发言时才启用“获取完整纠错评分”按钮。
- 用户点击“获取完整纠错评分”后，再调用完整评分接口生成纠错、建议和评分。
- 后端 `ai_client.py` 使用标准库 `urllib.request` 直接调用 DeepSeek API，保留 JSON 解析和 mock 兜底。
- `routes.py` 在真实 AI 失败时把错误摘要放入 warning，方便 Demo 前排查。

## 如何测试

```bash
python -m compileall backend\app
npm run build
git diff --check
```

手动测试：

- 切到逐句反馈模式，确认页面不再显示沉浸式 Qwen 实时语音对话面板。
- 未说话前，确认“获取完整纠错评分”按钮不可点击。
- 点击“录一句并对话”并说一句英文，确认 Qwen Realtime 会识别文本并先返回一句英文接话。
- AI 接话出现后，确认“获取完整纠错评分”按钮变为可点击。
- 点击“获取完整纠错评分”，确认右侧纠错反馈、学习建议和评分面板补齐。
- 切到沉浸式对话模式，确认 Qwen 实时语音对话入口仍可使用。
- 断开或移除真实模型 Key 时，确认页面会标明 mock 或本地兜底来源。
